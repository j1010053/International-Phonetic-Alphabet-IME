# IPA Windows 輸入法 — 系統規格書

**版本：** v0.2
**狀態：** 主要架構與範圍決策已定案；少數專案層面項目（時程、預算、維護歸屬等）待專案啟動時確認
**文件性質：** 活文件，隨設計細化與後續確認持續更新
**前次版本：** v0.1（架構部分定案；G、H 兩欄待決）

---

## 1. 專案概述

為國際音標（IPA, International Phonetic Alphabet）設計一套 Windows 輸入法，操作範式類似中文輸入法：使用者輸入英文字串，透過一對一或多對一的對應，輸出對應的 IPA Unicode 符號。本案以 Rime 引擎為基礎，定位為開源、個人/私人使用優先之專案。

---

## 2. 已定案決策

### 2.1 技術路線（引擎）
- 採用 **Rime 輸入法引擎**，Windows 前端為小狼毫（Weasel）。
- 理由：Rime 為表格驅動，正合 IPA「字串對應符號」模型；候選視窗、翻頁、選字、設定均為現成；開源、成熟、跨平台。
- 授權：Rime 採 GPLv3。因本案定位為開源（見 2.7），**無授權衝突**。

### 2.2 字符涵蓋範圍
- 第一階段涵蓋 **官方 IPA 表所列之全部字符**（基本子音、母音、附加符號 diacritics、超音段 suprasegmentals 等）。
- 碼表結構須 **預留擴充空間**，未來可增補（extIPA、官方修訂、使用者自訂符號）而不需重構。
- 前置需求：須由甲方指明依據之「官方 IPA 公布版本」，以鎖定第一階段字符清單。

### 2.3 編碼方案
- 主碼：採既有標準 **X-SAMPA**（互通、專家零學習成本）。
- 別名層（多對一）疊加於主碼之上：
  - 自訂助記別名（例：`sh` → ʃ、`th` → θ）。
  - 描述式名稱前綴別名：依 voicing–place–manner（子音）/ height–backness–rounding（母音）固定縮寫字典編碼，採固定欄位順序、voicing 預設 voiceless。
- 引擎實作：每個別名即碼表中一列，多列指向同一 Unicode 輸出（Rime 原生支援多對一）。
- 待細化設計：描述式縮寫消歧字典、母音/附加符號/塞擦音等不適用子音命名公式者的子方案。

### 2.4 組字與輸入行為
- 採 **注音式流程**：編碼累積於組字緩衝區，按 Enter 將整串上屏。
- 關閉唯一匹配自動上屏（對應 Rime `speller/auto_select = false`）。
- 候選確認：遇多碼或歧義時顯示候選供選字。
- 此設定同時解決前綴衝突（`s` 與 `sh`）與附加符號尚未疊加完成即被提前上屏的問題。
- 提醒：IPA 無詞句語言模型，行為為「逐符號選字 + 緩衝累積 + Enter 上屏」，非整句自動轉換。

### 2.5 平台與跨平台保留
- 必須支援：Windows 最新版本（當前 Win11）即可，**不要求舊版相容**。
- 保留跨平台可能：Rime 既有 macOS/Linux/Android 前端。所有方案資產（schema、碼表、資料來源）以引擎中立、可移植為設計原則；跨平台移植為「保留可能」，非首版交付。

### 2.6 顯示與字型
- IPA 字型隨附設為 **可選安裝元件**（建議 Charis SIL 或 Gentium，皆 OFL 授權可再散布）。
- **無特定 app 相容性保證需求**。
- 原則：輸入法只輸出 Unicode，不控制接收端字型；基本字母廣泛支援，疊加附加符號支援度隨 app 而異。提供「建議字型 + 套用說明」文件以校準期待。

### 2.7 授權與使用定位 *(本版定案)*
- **授權模式：開源。** 與 Rime GPLv3 相容，無散布限制。
- **使用定位：個人/私人使用優先。** 由此收斂以下部署選項至最輕量路徑：
  - 安裝範圍：每使用者安裝（不需管理員權限）。
  - 通路：不上架 Microsoft Store，採自家管道發佈。
  - 程式碼簽章：首版可省略（使用者安裝時接受未簽章警告）；未來對外擴大時再評估。

### 2.8 介面語言與國際化 *(本版定案)*
- **介面語言：中文優先。** 第一階段僅實際產出中文一套。
- **保留國際化（i18n）空間：** 設計上預留可換成英文或其他語言的能力，但非第一階段交付的工作量。
- **範圍界定：** 本案「介面語言」指 **我們可控制的輸入體驗與使用者文件**（方案名稱、F4 方案選單、開關標籤、候選符號說明、文件）。
- **Weasel 程式本體擱置：** 系統匣選單、「輸入法設定」對話框等為 Weasel 編譯內建之中文字串，無內建語言切換；本案 **不修改 Weasel 本體**，以一份對照說明帶過。（安裝精靈本身已內建中／英／繁三語，可選英文安裝。）
- **i18n 設計條款（必遵）：** 所有面向使用者的文字一律視為「可抽換資料」，集中於 **單一語言別主資料來源（master 資料表）**，再由產生器依指定語言產出 Rime schema、碼表與文件。新增語言 = 多填一個語言欄 + 重跑產生器，而非逐檔修改寫死字串。

## 3. 系統架構概覽

資料流以「單一主資料來源 → 產生器 → Rime 方案資產」為核心，這同時實現了碼表擴充（2.2）與 i18n（2.8）：

```
Master 主資料來源（單一事實來源）
  欄位：符號 | Unicode | X-SAMPA | 助記別名 | 描述式碼 | 說明(zh)[, en, ...]
        │
        ▼
   產生器（腳本，語言為參數）
        │
        ▼
Rime 方案資產：schema.yaml / default.yaml / 碼表(dict) / 使用者文件
        │
        ▼
   Rime 引擎（librime）
        │
        ▼
前端：Windows = Weasel ｜ 其他平台 = 對應 Rime 前端（保留）
        │
        ▼
   可選元件：IPA 字型安裝包（Charis SIL / Gentium）
```

## 4. 規模評估摘要

以單一具語音學基礎、願上手 Rime 設定之開發者估算（指示性，隨個人熟悉度浮動）：

- **MVP（驗證可用）：** X-SAMPA 主碼 + 注音式基本行為 + 官方基本符號 + 簡易文件，自家管道發佈。約 **2–3 週**。
- **完整 v1（目標）：** 加上助記與描述式別名及其子方案、附加符號疊加、候選說明、中文介面 + i18n 架構、可選字型、完整使用者編碼對照文件、測試。約 **6–8 週（1.5–2 個人月）**。
- **暫不納入：** 跨平台建置、自動更新、簽章/上架商店、extIPA、其他語言字串實際產出、為英文程式介面 fork Weasel。

成本重心不在工程，而在 **編碼方案設計** 與 **碼表資料的扎實度與核對**。

## 5. 建議開發順序

依相依關係排序；每步附簡短理由。MVP 切點標示於 P3 之後。

**P0　前置鎖定**
鎖定依據之官方 IPA 版本（2.2 前置需求）。確認後才能定字符清單。

**P1　編碼方案設計**
完成 X-SAMPA 對照、助記別名、描述式縮寫字典與消歧規則、母音/附加符號子方案，產出可審閱的編碼規格。
→ 理由：碼表與所有資產都依賴此方案定案，必須最先做。

**P2　建立 Master 主資料來源**
以單一資料表承載符號、Unicode、各層編碼與說明欄（先中文，預留 en 等欄位）。
→ 理由：這是 i18n 與擴充的載體（2.8 條款），須先於產生器存在。

**P3　產生器 + Rime schema POC**
寫產生器（依語言參數產出 dict/schema）；建立 Rime 方案並設定 `auto_select=false`、分隔符、表格翻譯、候選說明、注音式 Enter 上屏。以少量符號驗證行為。
→ 理由：先在小資料集驗證最高風險的「注音式行為 + 候選確認 + 疊加」，再投入全量資料。
　**——（MVP 切點：到此已有可用雛形）——**

**P4　碼表完整化**
將官方 IPA 全部符號、各層別名、中文說明補齊；以腳本預生成常見附加符號組合；重新產生全量方案。
→ 理由：方案、產生器、行為皆已驗證後，才放大資料量。

**P5　候選說明與顯示調校**
將說明接入候選視窗；以 `weasel.custom.yaml` 調整外觀。
→ 理由：核心可用後再處理體驗層。

**P6　可選字型元件**
封裝 Charis SIL / Gentium 為可選安裝。
→ 理由：獨立模組，可平行或後置。

**P7　打包與部署**
製作每使用者、自家管道之安裝/部署包；決定更新方式（個人使用預設手動更新）。
→ 理由：產品可用後再處理發佈。

**P8　測試**
功能測試（輸入/組字/候選/上屏）+ 少數常用 app 顯示抽測（輕量，無保證）。
→ 理由：發佈前驗證。

**P9　文件**
由 master 來源產生 **使用者編碼對照表**（自訂別名必須有教學）+ 安裝/使用說明（含 Weasel 原生中文選單之對照）。
→ 理由：內容多由 master 來源產生，可於後期定稿。

**P10　發布 v1。**

**未來迭代（保留）：** 其他語言字串產出、自動更新、簽章/上架、跨平台建置、extIPA。

## 6. 待確認事項（專案啟動時）

多數 G/H 項目已因「開源 + 個人使用」收斂至輕量預設；以下仍待確認：

- **官方 IPA 版本**（P0 前置，最優先）。
- **更新方式**：個人使用預設手動更新；若希望自動更新需額外投入派送機制。
- **無障礙需求**：是否需支援螢幕報讀、純鍵盤操作等（若使用者含機構/政府常為硬性要求）。
- **時程與預算邊界**：決定第一階段範圍與功能取捨。
- **維護與支援歸屬**：上線後維護、加新符號、使用者支援由誰負責。

## 7. 已知風險與假設

- 「官方 IPA 版本」未鎖定前，字符清單無法定案（P0 阻擋項）。
- 描述式編碼的縮寫消歧須嚴謹設計，否則影響可用性與可學習性。
- 疊加附加符號於部分 app 顯示可能不一致；因無特定 app 保證，屬已接受之已知限制。
- i18n 空間之有效性，取決於 2.8 設計條款是否確實落實（字串集中於 master 來源、由產生器產出）。
